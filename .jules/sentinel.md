## 2024-05-05 - Command Injection Risk in Non-Interactive Authorization
**Vulnerability:** The `requestNonInteractiveAuthorizationDecision` function in `src/cli/authorization/non-interactive.ts` passes user-configurable `cmd` string directly to `execa` with `shell: true`. This allows potential command injection if the config file is maliciously modified or if the `cmd` is constructed from untrusted sources, though it's typically loaded from a local config.
**Learning:** `execa` with `shell: true` evaluates the whole string as a shell command, which means shell metacharacters within the string are interpreted. When executing a command defined in a configuration file, it's safer to avoid the shell if possible, or parse the command into an executable and arguments. The memory constraints mention: "To mitigate command injection risks in tool authorization, the project avoids using `shell: true` in `execa` calls in `src/cli/authorization/non-interactive.ts`. The configuration supports an optional `args: string[]` field (in `ToolAuthorizationConfig`) for explicit arguments, while a fallback `splitCommand` utility in `src/core/utils/command-split.ts` safely parses command strings when `args` are not provided."
**Prevention:** Remove `shell: true`, update `ToolAuthorizationConfig` to support `args`, and implement/use `splitCommand` to parse the command string securely.
