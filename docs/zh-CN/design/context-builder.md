# 设计理念：上下文构建器 (Context Builder)

`ContextBuilder` 负责收集并收缩提供给 LLM 的信息。它遵循确定性的、基于规则的方法，以确保稳定性和可预测性。

## 1. 信息收集流水线

构建器遵循清晰的流水线：
1. **关键词提取**：从用户指令中进行简单提取。
2. **Ripgrep 搜索**：使用 `rg --json` 获取鲁棒的、机器可读的结果。
3. **Git Diff**：捕获当前工作区的更改。
4. **AST 分析**：识别主文件中的定义和引用。
5. **截断 (Truncation)**：将收集到的数据适配到字符预算中。

## 2. AST 增强上下文
构建器使用 Tree-sitter 对主文件进行轻量级 AST 分析：
- **定义 (Definitions)**：这些是函数、类或变量定义的位置。LLM 被指示在修改这些位置时要格外小心。
- **引用 (References)**：标记为 `↗️`。这些指示了符号被使用的位置，为 LLM 提供使用上下文，而无需引入整个文件。

## 3. 截断策略：顺序装载 (Pack-Until-Full)

我们不使用按比例截断，而是采用“顺序装载”策略：
- **优先级**：`主文本 (primaryText)` > `代码片段 (rgSnippets)` > `Git 差异 (gitDiff)`。
- **完整片段**：优先保留完整的代码片段，直到达到预算上限。
- **降噪**：如果超出预算，将丢弃 `gitDiff`，以避免无关更改干扰 LLM。

## 3. 确定性收缩 (Shrinking)

当验证失败时，上下文将根据以下内容收缩：
- **失败文件**：从验证输出（如堆栈追踪）中提取。
- **静态依赖**：有限度地扩展关联文件。
- **硬上限**：严格限制关联文件和片段的数量，防止上下文膨胀。

## 4. 启发式预算

所有限制都是基于字符而非 token 的。这使得系统：
- **模型无关**：在不同的 LLM 提供商之间表现一致。
- **可预测**：易于推导内存和资源使用情况。
- **稳定**：避免了本地分词 (tokenization) 的复杂性和开销。
