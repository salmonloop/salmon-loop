# 设计理念：执行限制 (Execution Limits)

SalmonLoop 通过有意设计的约束确保自动化补丁是确定、可评审且安全的。

## 1. 补丁安全限制

- **`最大文件更改数 (maxFilesChanged) = 2`**：强制更改的局部性。最可靠的修复通常是单文件或跨文件的微调。防止意外的大规模重构。
- **`最大 Diff 行数 (maxDiffLines) = 200`**：确保补丁保持可评审且语义集中。过大的 Diff 通常意味着任务需要进一步拆解。
- **`最大重试次数 (maxRetries) = 2`**：防止无限自纠错循环。当上下文或指令模糊时，鼓励带反馈的早期失败。

## 2. 上下文预算限制

- **`最大上下文字符数 (maxContextChars) = 30000`**：模型无关、可预测内存占用的启发式上限。能舒适地适配现代 LLM 的上下文窗口。
- **`最大主文件字符数 (maxPrimaryChars) = 12000`**：限制主文件内容以保持高信噪比，防止超大文件占据整个上下文。
- **`最小上下文保护值 (minContextChars) = 5000`**：防止重试期间上下文过度收缩，确保 LLM 始终有足够的信息进行判断。

## 3. 搜索与收缩限制

- **`最大关键词数 (maxKeywords) = 3`**：保持 ripgrep 搜索专注于特定指令，避免全库范围的噪音。
- **`最大关联文件数 (maxRelatedFiles) = 20`**：上下文收缩期间静态依赖的有限扩展，确保性能的确定性。
- **`收缩后最大片段数 (maxSnippetsAfterShrink) = 30`**：限制收缩后保留的代码片段数量，稳定上下文规模。
- **`最小片段字符数 (minSnippetChars) = 64`**：过滤掉无意义的代码碎片和部分 token。

## 4. 验证输出限制

- **`验证输出最大行数 (verifyOutputMaxLines) = 300`**：捕获关键失败信号（堆栈追踪、编译错误），而不撑爆上下文。

## 总结

这些限制强制执行了**有纪律的执行循环**。SalmonLoop 倾向于微小的、局部的更改和确定性的行为，而不是投机性的自动化，从而**最大化用户对自动化更改的信任**。
