# 设计理念：补丁循环 (Patch Loop)

SalmonLoop 的核心是一个形式化的执行循环，旨在实现“可预测、可解释、可嵌入”。

## 1. 形式化执行阶段

循环遵循严格的九阶段流程：
1. **预检 (PREFLIGHT)**：只读。检查环境安全性（Git 仓库、脏工作区检查）。
2. **上下文 (CONTEXT)**：只读。收集代码库上下文和目标文件内容。
3. **规划 (PLAN)**：只读。LLM 生成结构化的修改计划。
4. **补丁 (PATCH)**：只读。LLM 根据计划生成 unified diff。
5. **验证 (VALIDATE)**：只读。强制执行安全限制，执行模糊上下文匹配，并确保 diff 有效。
6. **应用 (APPLY)**：变更。**唯一**将更改写入磁盘的阶段。应用后会进行**深度 AST 校验**。
7. **验证 (VERIFY)**：只读。运行用户提供的检查（如测试）。
8. **回滚 (ROLLBACK)**：变更。如果验证失败，恢复文件系统状态。
9. **收缩 (SHRINK)**：只读。根据失败信号为下一次尝试缩减上下文。

## 2. 安全保证

- **原子性更改**：通过 `git apply` 应用更改。如果补丁的任何部分失败，则不会应用任何内容。
- **可靠回滚**：回滚是针对性的，基于当前尝试中更改的具体文件。如果遇到 Git 冲突或异常状态，系统会自动执行强制重置（`git reset --hard`）和清理（`git clean`），确保工作区始终恢复到干净的初始状态。
- **禁止文件操作**：禁止创建、删除和重命名文件，确保执行环境保持稳定且可逆。如果违反这些规则，验证阶段将提供详细的反馈。
- **模糊上下文匹配**：使用 Levenshtein 距离验证补丁上下文与实际文件内容，允许微小的空格或注释差异，同时确保结构正确性。

## 3. 可观测性

- **结构化日志**：每一步都记录其对应的 `执行阶段 (ExecutionPhase)`。
- **失败归因**：错误被显式链接到发生的阶段（例如 `failurePhase: VALIDATE`）。
- **历史追踪**：记录每次迭代，允许详细分析系统如何收敛到解决方案。
